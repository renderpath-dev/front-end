# Node.js 第 1 章运行时基础面试题

## 使用方法

先用“结论—机制—边界—证据”四步口述，再对照答案。答案以本章范围为边界，不把实现细节夸大为永久规范。

## 问题

1. Node.js 是语言、框架还是运行时？它解决了什么宿主能力问题？
2. 同一段 JavaScript 在浏览器和 Node.js 中为什么能得到不同结果？
3. V8 在 Node.js 中负责什么？它不负责什么？
4. libuv 与 V8 的职责边界是什么？
5. 事件循环“单线程”是否意味着 Node.js 进程只有一个线程？
6. 为什么 `setTimeout(callback, 0)` 不保证立即执行？
7. 顶层 `setTimeout(..., 0)` 与 `setImmediate()` 的顺序为什么不能硬编码？
8. `process.nextTick()` 与 Promise 微任务有什么风险差异？
9. 为什么异步 I/O 可以提升并发能力，却不能自动解决 CPU 密集任务？
10. 网络 I/O 和文件 I/O 是否都通过 libuv 线程池？
11. `process.exitCode = 1` 与 `process.exit(1)` 的工程差异是什么？
12. 注册 `SIGINT` 监听器后，为什么进程可能不再自动退出？
13. `process.argv` 前两个元素是什么？
14. `process.env` 的类型和生命周期边界是什么？
15. 如何用 `monitorEventLoopDelay()` 判断 JavaScript 线程被阻塞？
16. `os.availableParallelism()` 能否证明事件循环有多个线程？
17. TypeScript 能否保证事件循环回调顺序？
18. 如何在真实 HTTP 服务中识别阻塞风险？

## 参考答案

### 1

Node.js 是 JavaScript 运行时，不是语言，也不是 Web 框架。它内嵌 V8，并通过 Node 核心 API、原生绑定、libuv 和操作系统向 JavaScript 暴露文件、网络、进程、计时器等宿主能力。

### 2

ECMAScript 只规定语言核心。`window`、DOM、`process`、文件系统等来自不同宿主。语法可以相同，但可访问的全局对象、模块加载器、权限和生命周期不同。

### 3

V8 解析、编译和执行 JavaScript，管理对象、堆与垃圾回收。它不定义 Node 的文件系统 API、HTTP 服务器、事件循环阶段或操作系统信号。

### 4

V8 执行 JavaScript；Node 的原生绑定把 JavaScript API 连接到 C/C++；libuv 提供事件循环、跨平台异步抽象、线程池和部分系统能力。网络与文件操作最终仍由操作系统完成。

### 5

不是。通常只有一个线程执行用户 JavaScript 回调，但 V8、libuv 线程池、垃圾回收和操作系统都可能使用其他线程。不要把“一个 JavaScript 事件循环线程”缩写成“进程只有一个线程”。

### 6

零毫秒是阈值，不是执行期限。回调必须等当前调用栈结束，并等事件循环进入可处理计时器的时机；阻塞代码、其他回调和系统调度都会增加延迟。

### 7

它们进入不同调度机制；顶层运行时所处的循环状态和系统时序会影响先后。只有把场景限制到 I/O 回调等明确上下文，才能讨论可重复的阶段关系。

### 8

两者都在阶段回调之间运行，但 `process.nextTick()` 队列具有更高的 Node 特有优先级。递归添加 nextTick 会持续阻止事件循环进入 I/O 阶段；Promise 递归也可能造成微任务饥饿，但队列和 API 边界不同。

### 9

异步 I/O 把“等待资源完成”的时间交给 OS 或 libuv，让 JavaScript 线程处理其他回调。CPU 密集 JavaScript 本身仍占用该线程，必须拆分、限制、移到 Worker Threads/进程或使用更合适的原生能力。

### 10

不是。网络 I/O 通常通过操作系统的非阻塞轮询机制；很多文件系统操作、`getaddrinfo`、部分加密和用户排队工作使用 libuv 线程池。具体 API 必须查官方文档。

### 11

设置 `exitCode` 后进程自然退出，已排队的输出和回调仍有机会执行；`process.exit()` 强制终止，可能截断异步 stdout/stderr。正常错误路径优先使用 `exitCode`。

### 12

Node 的默认终止行为被监听器替代。处理器需要停止资源、设置退出码，并让事件循环自然变空；若仍有服务器或计时器引用，进程会继续存活。

### 13

`process.argv[0]` 是 Node 可执行文件路径，`process.argv[1]` 通常是入口脚本路径，之后才是用户参数。

### 14

读取结果是字符串或 `undefined`。修改只影响当前进程环境及随后创建的子进程，不会回写父 shell；平台和工作线程还存在大小写及副本差异。

### 15

启用延迟直方图，预热至少几个采样周期，运行待观察工作，再等待采样并关闭。把纳秒转换成毫秒，比较 `max`、均值和高百分位；长阻塞通常产生明显峰值。

### 16

不能。它给出应用可使用的默认并行度估计，用于规划 worker 数量；它不改变事件循环执行用户 JavaScript 的线程模型。

### 17

不能。TypeScript 只在编译期检查类型并擦除为 JavaScript。回调时序由 Node 运行时、V8 微任务处理、libuv、操作系统和实际负载决定。

### 18

检查同步 API、无界循环、大 JSON 解析/序列化、复杂正则、递归 nextTick、过大的单次计算；同时测量事件循环延迟和请求尾延迟，并用阻塞与异步对照实验确认因果链。
